Ch8: Forms

Every HTML form is delimited by the

and
tags, among which the elements that make up the form must be found, as well as any other HTML element that we need: images, links, etc. In an HTML document there can be as many forms as needed, but they cannot be nested.

The attributes of the FORM tag are:

  • ACTION specifies the URL of the CGI script that should process the data collected by the form. This is the link between the form and the CGI script.
  • METHOD determines the method to use to access the URL specified in the ACTION field. There are two methods, GET (the default) and POST (based on HTTP methods). In general cases, GET will be used when we have very simple forms, and POST when the forms have many fields, since GET might not be able to pass all the data to the server. METHOD="GET"

The form information is added to the end of the requested URL, behind a question mark ,`?'. This is what most web browsers do. The CGI script will receive it encoded in the environment variable QUERY_STRING. The main problem with this method is that it has length restrictions because many operating systems limit the name of a file to 256 characters, for example, and the Web server treats the URL, including the query, as if it were a type of file. . Another disadvantage is that the URL, including the query, is stored in the access logs of the servers, which may not be recommended if the query information is to be kept secret. METHOD="POST"

The CGI script receives the information from the form through its standard input. The server does not set the end-of-file mark, EOF. Instead, and to know the length of the input data, the bytes are specified in the CONTENT_LENGTH variable.

Using FormData Objects Effectively

Instead of individually going to each one of the form elements when you want to bundle the data up and send it to the server you can use the form data object and very quickly and easily do parse the forms and gather up all the data.

For the form data object to work, you should have the name attribute inside your inputs. When you just have an ID, it appears to work because there are no errors or complaints, but when you look at the form data object after you've gathered everything, it will be empty because none of the inputs will come through unless they have a name attribute. ID works great for JavaScript and CSS, but to work inside the form data object, you must have a name attribute.

MDN: Client-Side Form Validation

"Before submitting data to the server, it is important to ensure all required form controls are filled out, in the correct format. This is called client-side form validation and helps ensure data submitted matches the requirements set forth in the various form controls. "

Form Validation examples:

  • This field is required" (You can't leave this field blank).
  • "Please enter your phone number in the format xxx-xxxx" (A specific data format is required for it to be considered valid).
  • "Please enter a valid email address" (the data you entered is not in the right format).

    • Validation Attributes

  • required: Specifies whether a form field needs to be filled in before the form can be submitted.
  • minlength and maxlength: Specifies the minimum and maximum length of textual data (strings)
  • min and max: Specifies the minimum and maximum values of numerical input types
  • type: Specifies whether the data needs to be a number, and email address, or some other specific preset type.
  • pattern: Specifies a regular expression that defines a pattern the entered data needs to follow.
